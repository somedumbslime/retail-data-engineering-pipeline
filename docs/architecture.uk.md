# Нотатки По Архітектурі

## Навіщо Потрібен Staging

Staging — це технічний буфер між OLTP і аналітичними шарами.
Він ізолює продакшн-транзакції від важких аналітичних читань і дає стабільний проміжний шар для ETL.

Переваги staging:
- зберігає сирі скопійовані дані близькими до формату джерела
- дозволяє повторювані backfill і reprocessing
- зменшує пряме навантаження на OLTP

## Навіщо Потрібен DWH

DWH оптимізований для аналітики та історичної звітності, а не для транзакційного запису.
У цьому проєкті DWH зберігає узгоджені виміри (`dim_users`, `dim_products`) і вимірювані події (`fact_sales`).

Переваги DWH:
- швидші агрегувальні запити
- простіша звітна модель (star schema)
- контрольовані бізнес-визначення KPI

## OLTP vs DWH

OLTP:
- підтримує операції застосунку в реальному часі (insert/update/delete)
- нормалізована схема
- оптимізований для великої кількості малих транзакцій

DWH:
- підтримує аналітику (select, group by, аналіз трендів)
- денормалізована або вимірна схема
- оптимізований для сканування та агрегування

## Потік Даних

`OLTP -> Staging -> DWH -> Data Marts`

- OLTP: операційні вихідні таблиці
- Staging: сиро скопійовані дані для контрольованої обробки
- DWH: вимірна модель для аналітики
- Marts: бізнес-орієнтовані агреговані вітрини
